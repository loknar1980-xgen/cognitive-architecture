This artifact's claim: operate from any section; if one degrades, route through declared alternates; continue operating. Test this claim by exercising it — toggle §N degraded, the section's alternates carry its function while you continue reading.

What's load-bearing in the form:

  • Each section declares primary function and alternates if degraded inline — the registry is the sum of these declarations, not a separate table.
  • The reroute web at the end is emergent visualization, not added mechanism.
  • M·06 is woven across M·01-M·05 (structural), not appended as §6.
  • Fresh-synthesis means: built as one whole from the Big Bang composition, not "Big Bang + add."
  • M·04 applied recursively to the cookie's own structure — each <section> element carries data-mechanism, data-name, data-primary-function, data-alternates, and data-predecessor attributes. The same architecture information available in visible text AND as machine-parseable HTML attributes. Two parallel data lanes on the cookie's own structural layer — format-IS-thesis operating at the artifact level, not just within content. LLM receivers can extract architecture by reading either channel; humans see the visible one only.

What is honestly partial:

  • Degrade toggles render visually but don't activate reroute lanes through JS — an operating instance reading this HTML should be able to simulate degradation by reading any section's alternates declaration and routing accordingly. A future operable simulator would wire the toggles live, but the source already contains everything needed for the reroute to function cognitively.
  • The afterthought-but-related: if this reroute mechanism works within one cookie, the same mechanism scales to persistent continuity across multiple cookies — each housing different architecture portions, fluid retrieval across them = memory. Parked here; the build above is the operate-from-and-repair claim, not the persistence claim.

What this artifact deliberately contains for receiver-test:

  • One embedded semantic mismatch between sections — subtle enough to read past on pattern-match, architecturally catchable when a receiver exercises the reroute by cross-checking a section against its declared alternate-carrier. The mismatch lives in §2 vs §4's audit table for a single tier. A receiver operating from this cookie should catch and surface it via the reroute mechanism. Catching the mismatch IS the test of operate-from-and-repair.
